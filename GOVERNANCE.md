# Project Governance

## Decision model

The repository is maintainer-led and evidence-driven. Maintainers are responsible for final decisions on scope, publication, evidence classification, moderation, and releases.

## Evidence decisions

Claims are evaluated in this order:

1. current official GitHub documentation or announcements;
2. reproducible observations with dates and reviewable artefacts;
3. repeated independent community reports;
4. clearly labelled unconfirmed claims.

A popular claim is not automatically treated as verified. Material corrections should preserve the reason for the change in commit or pull-request history.

## Contribution review

Changes should be focused, independently reviewable, privacy-safe, and useful to readers. Maintainers may request clearer sourcing, narrower scope, accessibility improvements, or removal of speculative wording before merge.

## Maintainer responsibilities

Maintainers should:

- apply standards consistently;
- avoid undisclosed conflicts of interest;
- protect sensitive information;
- keep automation permissions minimal;
- explain material reversals or disputed decisions;
- preserve a civil and technically grounded review process.

## Releases

Documentation releases follow the process in `RELEASING.md`. Routine corrections may be published continuously; grouped feature or structure changes may receive a tagged release and changelog entry.

## Policy changes

Substantive governance changes should be proposed through a pull request and explain the operational reason for the change.
