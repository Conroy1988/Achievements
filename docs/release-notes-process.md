# Release Notes Process

Material updates should be summarised in release notes or a changelog entry.

Include:

- the achievements or tiers affected;
- whether a claim was added, corrected, retired, or reclassified;
- the evidence level before and after the change;
- links to supporting documentation or reproduction records;
- any known uncertainty or follow-up work.

Avoid announcing community-observed behaviour as an official GitHub guarantee. Release notes should help readers identify what changed without requiring them to inspect every commit.
